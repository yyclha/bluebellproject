# Milvus Native BM25 Design

## Goal

Move lexical retrieval from the Go in-memory BM25 cache into Milvus native full text search, then use Milvus hybrid search to combine dense embedding recall and BM25 sparse recall.

## Context

The current RAG collection stores chunk metadata and a dense `embedding` vector. Lexical recall is handled in Go by loading MySQL RAG chunks, building an in-memory BM25 index, and merging dense and lexical results in business logic. Milvus 2.6 supports BM25 functions that generate sparse vectors from a text field, plus hybrid search across dense and sparse vector fields.

## Architecture

The Milvus collection will contain the existing chunk fields plus a new sparse vector field:

- `chunk_id`: primary key.
- `post_id`: source post ID.
- `chunk_index`: chunk ordinal in the post.
- `chunk_text`: chunk text, with analyzer enabled.
- `embedding`: dense embedding vector.
- `sparse_embedding`: sparse vector generated by a BM25 function from `chunk_text`.

Collection creation will register a BM25 function from `chunk_text` to `sparse_embedding`. Dense search keeps using the configured vector index. Sparse search uses a sparse inverted index with BM25 metric. Search will move from separate Milvus dense search plus local BM25 to one Milvus hybrid search call with RRF ranking.

## Migration

This change requires Milvus server and Go SDK versions that support full text search, BM25 functions, sparse vector fields, and hybrid search. The application commonly runs on Windows while Milvus runs separately in a VM Docker environment, so the application config should point `milvus.address` at the VM IP and port 19530. Existing RAG collections should not be reused because the schema changes. Use a new collection name such as `post_rag_chunk_hybrid_1024`, then run the existing RAG reindex endpoint or command to repopulate chunks.

## Behavior

Indexing still computes dense embeddings in the application and writes chunk text plus dense vectors to Milvus. Milvus generates sparse BM25 vectors from chunk text internally. Querying still embeds the query once for dense search, and also passes the raw query text to the sparse BM25 request. Hybrid search returns chunk-level hits, then the DAO deduplicates by post ID to preserve the current response shape.

## Error Handling

If Milvus is disabled or uninitialized, RAG search continues to return the existing Milvus-disabled error. If collection schema validation finds an old dense-only collection, startup fails with a schema incompatibility message and instructs the operator to use a new collection or rebuild. Reindexing remains the recovery path after schema changes.

## Testing

Unit tests should cover schema validation requirements and search result conversion where possible without a live Milvus server. Full integration with BM25 function and hybrid search requires a Milvus 2.6+ instance and will be documented as a manual verification step.
